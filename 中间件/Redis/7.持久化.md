# Redis持久化机制

**持久化（Persistence）** 是 Redis 从“单纯的缓存”进化为“可靠的数据库”的关键。

Redis 主要提供两种持久化方式：**RDB（快照）** 和 **AOF（追加日志）**，以及在现代版本（4.0+）中默认推荐使用的 **混合持久化**。

---

## 1. RDB (Redis Database) —— 内存快照

RDB 是在指定的时间间隔内，将内存中的全量数据集生成一个二进制文件（通常是 `dump.rdb`）保存到磁盘上。

**工作原理**：Redis 会 `fork` 一个子进程。子进程将当前时刻的内存数据写入一个临时文件，写入完成后替换旧的 RDB 文件。

**优点**：

* **恢复极快**：直接将二进制数据加载回内存，适合大规模数据恢复。
* **性能影响小**：主进程不进行 IO 操作，保持了高吞吐量。

**缺点**：


* **数据丢失风险高**：如果两次快照之间宕机，会丢失最后一次快照后的所有数据。
* **Fork 耗时**：如果数据集巨大，`fork` 子进程时可能会导致毫秒级的请求停顿。

---

## 2. AOF (Append Only File) —— 命令追加

AOF 以独立日志的方式记录每次写命令，重启时通过重新执行这些命令来恢复数据。

**写回策略 (appendfsync)**：

* `always`：每条命令都同步到磁盘（最安全，性能最差）。
* `everysec`：每秒同步一次（**默认推荐**，性能与安全的平衡）。
* `no`：操作系统控制同步（性能最好，不安全）。

**AOF 重写 (Rewrite)**：


* 为了防止 AOF 文件无限膨胀，Redis 会定期重写。它不读取旧文件，而是直接读取当前内存数据生成一套最小化的指令集（例如：将 100 次 `INCR` 合并为一条 `SET`）。

**优点**：数据安全性高，丢失率可控。

**缺点**：文件体积大，恢复速度比 RDB 慢。

---

## 3. 混合持久化 (RDB + AOF) —— 现代标配

在 Redis 4.0 之后，引入了 `aof-use-rdb-preamble` 配置。

* **逻辑**：在 AOF 重写时，先将当前的内存状态以 RDB 的格式写入 AOF 文件的头部，之后产生的写命令再以 AOF 格式追加。
* **效果**：结合了 RDB 的**加载快**和 AOF 的**数据完整性**。

---

## 4. 核心对比与选型建议

| 维度 | RDB | AOF | 混合持久化 |
| --- | --- | --- | --- |
| **持久化方式** | 定时全量备份 | 实时/秒级指令追加 | RDB 头 + AOF 尾 |
| **恢复速度** | 很快 | 慢 | 快 |
| **数据安全性** | 较低（丢分钟级数据） | 高（丢秒级数据） | 高 |
| **文件大小** | 小（二进制压缩） | 大（文本格式） | 中 |

---

## 5. 源码层面的持久化实现 (简化版)

在 Redis 源码中，持久化主要由以下函数入口控制：

* **RDB**: `rdbSave()` (同步) 或 `rdbSaveBackground()` (异步)。
* **AOF**: `flushAppendOnlyFile()`。

```c
/* 伪代码演示 AOF 写入逻辑 */
void call(client *c, int flags) {
    // 1. 执行命令
    c->cmd->proc(c);
    // 2. 如果开启了 AOF，将命令追加到 aof_buf 缓冲区
    if (server.aof_state == AOF_ON) {
        feedAppendOnlyFile(c->cmd, c->db->id, c->argv, c->argc);
    }
}
```
