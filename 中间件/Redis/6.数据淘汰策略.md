# 数据淘汰策略
**内存是 Redis 最宝贵的资源**。当 Redis 内存使用达到 `maxmemory` 限制时，它必须决定是拒绝写入还是“牺牲”掉一部分旧数据。

这套决策逻辑就是 **数据淘汰策略（Eviction Policies）**。

Redis 目前提供 **8 种** 主要的淘汰策略，我们可以将其归纳为三个维度。

---

## 1. 淘汰策略分类图解

| 维度 | 策略名称 | 行为描述 |
| --- | --- | --- |
| **不淘汰** | **`noeviction`** | **默认策略**。内存满时，写入操作报错，仅允许读和删除。 |
| **设置了过期时间的 (volatile)** | **`volatile-lru`** | 在设置了过期时间的键中，淘汰**最近最少使用**的。 |
|  | **`volatile-lfu`** | 在设置了过期时间的键中，淘汰**访问频率最低**的。 |
|  | **`volatile-ttl`** | 在设置了过期时间的键中，优先淘汰**剩余寿命最短**的。 |
|  | **`volatile-random`** | 在设置了过期时间的键中，**随机**挑选淘汰。 |
| **所有键 (allkeys)** | **`allkeys-lru`** | 在所有键中，淘汰**最近最少使用**的（最常用）。 |
|  | **`allkeys-lfu`** | 在所有键中，淘汰**访问频率最低**的。 |
|  | **`allkeys-random`** | 在所有键中，**随机**挑选淘汰。 |

作为内存数据库，出于对性能和内存的考虑，Redis 的淘汰算法实际实现上并非针对所有 key，而是抽样一小部分并且从中选出被淘汰的 key。

---

## 2. 核心算法深度解析

#### LRU (Least Recently Used) —— 最近最少使用

* **逻辑**：关注**时间点**。如果一个数据很久没被访问，它被淘汰的概率最高。
* **Redis 实现**：Redis 并没有使用真正的链表维护 LRU（太占内存），而是采用**近似 LRU 算法**。它在 `redisObject` 中记录 `lru` 时间戳，每次淘汰时随机采样 N 个键，然后踢掉其中最旧的。

#### LFU (Least Frequently Used) —— 最不经常使用

* **逻辑**：关注**频率**。如果一个数据 1 小时前访问了 100 次，另一个数据 1 分钟前访问了 1 次，LRU 会淘汰前者，而 LFU 会保留前者。
* **Redis 实现**：将 `lru` 字段拆分，高位记录分钟级时间戳，低位记录 **8bit 计数器**（对数计数器，最大可表示百万次访问）。

---

## 3. 如何选择策略

在你的 **Spring Boot + K8s** 环境中，选择策略通常遵循以下逻辑：

1. **作为纯缓存使用（无持久化需求）**：
* 推荐 **`allkeys-lru`**。这是最通用的方案，能确保热点数据留在内存。


2. **存在明显的长尾访问分布**：
* 推荐 **`allkeys-lfu`**。如果你希望某些极高频访问的数据（如全局配置）永不消失，哪怕它最近几秒没被访问。


3. **既做缓存又做分布式锁/持久化存储**：
* 推荐 **`volatile-lru`**。这样 Redis 只会动你设置了 `expire` 的缓存数据，而不会误删你的分布式锁或永久存储的配置。


4. **业务严谨，宁愿报错也不允许丢数据**：
* 使用默认的 **`noeviction`**，但一定要配合完善的监控告警。
