# Redis与Memcached

## 1. 对比表

| 特性 | Redis (7.x / 8.x) | Memcached |
| --- | --- | --- |
| **数据类型** | **丰富** (String, List, Hash, ZSet, Stream, JSON等) | **单一** (仅支持 String/Blob) |
| **线程模型** | 主线程单线程 (现代版本支持多线程 I/O) | **多线程** (原生多线程，更易榨干多核 CPU) |
| **持久化** | **支持** (RDB 快照, AOF 日志) | **不支持** (全内存，断电即失) |
| **内存管理** | 动态分配 (jemalloc)，可能存在碎片 | **Slab Allocation** (预分配内存，几乎无碎片) |
| **高可用** | 原生集群 (Cluster)、主从 (Sentinel) | 依赖客户端分片 (Client-side sharding) |
| **典型延迟** | 微秒级 (通常 < 1ms) | 微秒级 (高并发下略优于 Redis) |

---

## 2. 深度技术差异

### A. 多线程 vs 单线程

* **Memcached** 是天生的多线程模型。如果你有一台 64 核的服务器，Memcached 可以通过多线程并行处理请求。
* **Redis** 虽然在 6.0 后引入了多线程 I/O 来处理网络读写，但**执行命令**依然是单线程的。
* **专家结论**：对于超高并发的**简单 GET/SET** 操作，Memcached 的吞吐量上限通常高于单实例 Redis。

### B. 数据操作的灵活性

* **Memcached** 的操作逻辑是：`Get整个对象 -> 在应用层修改 -> Set回缓存`。这涉及频繁的序列化和网络传输。
* **Redis** 支持**服务端计算**。比如你想给用户的积分加 1，直接调用 `INCR`；或者更新 Hash 里的一个字段。
* **结论**：在Spring Boot 应用中，Redis 能显著减少网络 I/O 损耗，因为“计算被移动到了数据所在的地方”。

### C. 内存分配策略

* **Memcached** 使用 **Slab Allocation**：启动时就把内存切成固定大小的块（Slab）。
  * 优点是**没有内存碎片**，缺点是如果对象大小分布不均，会造成浪费，内存利用率不高。

* **Redis** 依赖 **malloc/jemalloc** 动态分配。
  * 优点是**内存利用率极高**，缺点是在长时间运行、频繁删改大对象后，可能产生内存碎片（需要通过 `activedefrag` 整理）。
  * 数据不是都一直存储在内存中，可以将一些旧数据交换到磁盘。


## 建议

* **选 Memcached 的理由**：业务极其简单（全是 Key-Value 字符串），且并发量巨大（每秒数百万次请求），极其在意单机吞吐量。
* **选 Redis 的理由**：**95% 的互联网场景。** 你需要排行榜、分布式锁、会话持久化、或者是像 JSON/Stream 这样复杂的数据处理能力。
