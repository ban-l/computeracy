# 数据类型

## 1. String（字符串）

这是最基础的类型，它是 **二进制安全** 的，可以存文本和图片。

- **底层实现**：**SDS (Simple Dynamic String)**。
  - 相比 C 语言原生字符串，SDS 会记录长度（获取长度 `O(1)`），且支持空间预分配，防止缓冲区溢出。
- **常用场景**：缓存、计数器（`INCR`）、分布式锁（`SETNX`）。

**常用命令：**

- **`SET key value [EX seconds] [NX|XX]`**：设置值。`NX` 表示键不存在时才设置（分布式锁核心），`EX` 设置秒级过期。
- **`GET key`**：获取值。
- **`INCR / DECR key`**：原子递增/递减 1。常用于计数器、限流。
- **`INCRBY / DECRBY key increment`**：原子增减指定数值。
- **`MSET / MGET`**：批量设置/获取，减少网络 RTT（往返时间）。
- **`SETEX key seconds value`**：设置值的同时指定过期时间。

## 2. List（列表）

一个双向链表，支持从左右两端推入或弹出元素。

- **底层实现**：在现代 Redis（3.2+）中统一使用 **quicklist**（快速列表）。
  - 它是双向链表和 **ziplist**（压缩列表）的结合体，既保证了插入效率，又减少了内存碎片。
- **常用场景**：消息队列（`LPUSH` + `BRPOP`）、最新消息排行。

**常用命令：**

- **`LPUSH / RPUSH key value`**：从左侧/右侧插入元素。
- **`LPOP / RPOP key`**：从左侧/右侧弹出一个元素。
- **`LRANGE key start stop`**：获取指定范围内的元素（如 `0 -1` 获取全部）。
- **`LLEN key`**：获取列表长度。
- **`BRPOP / BLPOP key timeout`**：**阻塞式**弹出。如果列表为空则等待，直到有数据或超时（消息队列常用）。
- **`LREM key count value`**：移除指定数量的某个值。

## 3. Hash（哈希）

键值对集合，非常适合存储对象（如用户信息）。

- **底层实现**：
  - 当数据量小时，使用 **ziplist**（内存紧凑）。
  - 当数据量大或字段长时，转换为 **hashtable**（哈希表，`O(1)` 查询）。
- **常用场景**：存储对象、购物车信息。

**常用命令：**

- **`HSET key field value`**：设置哈希表中的字段。
- **`HGET key field`**：获取指定字段的值。
- **`HGETALL key`**：获取所有字段和值（慎用，字段过多会阻塞）。
- **`HDEL key field`**：删除一个或多个字段。
- **`HINCRBY key field increment`**：对哈希字段进行原子增减。
- **`HEXISTS key field`**：检查字段是否存在。

## 4. Set（集合）

无序、唯一的字符串集合。

- **底层实现**：
  - 如果全是整数且数量少，使用 **intset**（整数集合）。
  - 否则使用 **hashtable**（其值 Value 均为 NULL）。
- **常用场景**：去重、社交关系（共同关注、求交集/并集）。

**常用命令：**

- **`SADD key member`**：向集合添加元素。
- **`SREM key member`**：移除元素。
- **`SMEMBERS key`**：获取集合所有成员（注意大数据量风险）。
- **`SISMEMBER key member`**：判断元素是否存在（`O(1)` 效率极高）。
- **`SCARD key`**：获取集合成员总数。
- **`SINTER / SUNION / SDIFF`**：求两个集合的 **交集 / 并集 / 差集**（常用于好友推荐、共同爱好）。

## 5. ZSet（有序集合）

Redis 最具特色的结构。每个成员关联一个 `score`（分数），按分数从小到大排序。

- **底层实现**：**跳跃表 (SkipList)** + **dict**。
  - 跳跃表通过多层索引实现了平均 `O(\log N)` 的查询效率，性能媲美平衡树，但实现更简单。
- **常用场景**：排行榜、带权重的任务队列。

**常用命令：**

- **`ZADD key score member`**：添加成员并指定分数。
- **`ZSCORE key member`**：获取成员的分数。
- **`ZRANGE key start stop [WITHSCORES]`**：按分数**从小到大**返回成员。
- **`ZREVRANGE key start stop [WITHSCORES]`**：按分数**从大到小**返回（排行榜常用）。
- **`ZRANGEBYSCORE key min max`**：返回分数在指定范围内的成员。
- **`ZREM key member`**：移除成员。
- **`ZCARD key`**：获取成员总数。

## 6. 高级/特殊数据类型

除了五大基本类型，Redis 还提供了一些应对极限场景的黑科技：

| **数据类型**      | **特性**                               | **典型应用**                              |
| ----------------- | -------------------------------------- | ----------------------------------------- |
| **Bitmap**        | 位图，通过位操作（0/1）存储信息        | 极低内存实现“用户签到”、“日活统计”        |
| **HyperLogLog**   | 基数统计，极小内存（12KB）统计海量数据 | 统计 UV（独立访客），允许 `0.81\%` 的误差 |
| **Geospatial**    | 地理位置信息（基于 ZSet 实现）         | “附近的人”、距离计算                      |
| **Stream** (5.0+) | 持久化的消息流，支持消费者组           | 真正生产级别的消息队列方案                |

## 深度建议：

**1. 拒绝 `O(N`) 命令**

在生产环境中，尽量避免使用 `KEYS *`、`HGETALL`、`SMEMBERS` 等命令。它们会阻塞 Redis 唯一的处理线程。

**2. 关于大 Key (BigKey) 的处理**

如果你发现某个 Hash 或 List 的元素达到了几万个，删除它时请使用 **`UNLINK`** 而不是 `DEL`。

`UNLINK` 会在后台线程异步释放内存。

## 内存淘汰与对象共享

**Redis 对象头（redisObject）**。每个 Key-Value 实际上都有一个对象头，记录了：

- **type**: 逻辑类型（如 String）。
- **encoding**: 物理实现（如 SDS 还是 int）。
- **lru/lfu**: 记录访问时间，用于内存淘汰策略。
