# 常见问题

## 一、脑裂（Sentinel Mode）

### 1. 什么是“脑裂”？

想象一下，如果一个人突然拥有了两个大脑，且每个大脑都向身体下达不同的指令，系统必然崩溃。

在 Redis 哨兵架构中，所谓的“脑裂”，是指由于发生严重的**网络分区（Network Partition）**，导致原本正常的集群分裂成了两个或多个独立的网络孤岛。

最终结果是：**系统里同时出现了两个（或多个）都认为自己是 Master（主节点）的节点，并且它们都在接收客户端的写请求。**

### 2. 脑裂是如何导致数据“灰飞烟灭”的？（灾难演进时间线）

为了让你看清全貌，我们来推演一次典型的脑裂事故：

1. **和平时期：** 你有一个 1 主 2 从的架构，外加 3 个哨兵节点。应用客户端正常连接主节点进行数据读写。
2. **网络斩断（脑裂发生）：** 突然发生机房网络故障。老主节点（我们叫它 Master A）和一部分应用客户端被困在了“网络孤岛 A”中；而 3 个哨兵、2 个从节点以及另一部分应用客户端则在“网络孤岛 B”中。
3. **哨兵夺权：** 孤岛 B 中的哨兵们发现 Master A 连不上了（客观下线 ODOWN），于是立刻走故障转移流程，将其中一个从节点晋升为**新主节点（Master B）**。
4. **各自为政：** 此时灾难开始了。孤岛 A 中的老主节点 Master A 其实没死，孤岛 A 里的客户端还在源源不断地向它写入数据；与此同时，孤岛 B 里的客户端也在向新主节点 Master B 写入数据。**双主并存局面形成**。
5. **残酷的重组（数据丢失）：** 网络终于修复了！哨兵集群重新联系上了老主节点 Master A。由于集群中只能有一个真正的 Master，哨兵会毫不留情地将老主节点 Master A **强制降级为从节点**。

   * **致命打击：** 作为从节点，Master A 必须清空自己的所有历史数据，去和新主节点 Master B 进行全量数据同步。

   * **结果：** 在网络断开的这段时间里，客户端写给老主节点 Master A 的所有数据，**全部被抹杀，永久丢失！**

---

### 3. 如何解决（或缓解）脑裂带来的数据丢失？

Redis 的设计哲学是偏向 AP（可用性和分区容错性）的，它默认优先保证可用性。

要解决脑裂导致的数据丢失，哨兵自己做不到，我们需要在 **Redis 节点的 `redis.conf` 配置文件**中做文章。

核心防范思路是：**断掉老主节点的“笔”，一旦它发现自己成了光杆司令，就拒绝接收任何写请求。**

这需要配合以下两个极其关键的配置参数（在旧版本 Redis 中，`replicas` 被称为 `slaves`）：

* **`min-replicas-to-write <N>`：** 主节点必须至少拥有 N 个处于正常连接状态的从节点，才能允许执行写操作。
* **`min-replicas-max-lag <M>`：** 正常连接的定义是，从节点向主节点发送 ACK 确认消息的延迟不能超过 M 秒。

### 4. 防御机制是如何生效的？

假设我们配置了 `min-replicas-to-write 1` 和 `min-replicas-max-lag 10`。

当脑裂发生，老主节点 Master A 被隔离在网络孤岛中时，它与所有从节点的连接必然中断。
10 秒钟过去后，老主节点会发现：“我没有任何一个延迟在 10 秒内的健康从节点了”。
此时，老主节点就会**主动拒绝**所有来自客户端的写请求（返回错误）。

这样一来，在网络恢复后的清空同步阶段，最多也只会丢失网络断开最初那 10 秒钟的数据，从而把损失降到了最低。

> **提示：** 
>
> 开启这两个配置会牺牲一部分 Redis 的高可用性（如果从节点真的全挂了，主节点也会跟着停止写服务）。但在对数据完整性要求极高的金融或电商交易核心场景中，这种“宁可写失败，也不要丢数据”的取舍是非常必要的。

## 二、分片

分片是将**数据划分为多个部分**的方法，可以将**数据存储**到**多台机器**里面。

假设有 4 个 Redis 实例 R0，R1，R2，R3，还有很多表示用户的键 user:1，user:2，... ，有不同的方式来选择一个指定的键存储在哪个实例中。

- **范围分片**：例如用户 id 从 0~1000 的存储到实例 R0 中，用户 id 从 1001~2000 的存储到实例 R1 中等等。
  - 但是这样要维护一张映射范围表，维护操作代价很高。
- **哈希分片**：使用 CRC32 哈希函数将键转换为一个数字，再对实例数量求模，就能知道应该存储的实例。

根据执行分片的位置，可以分为三种分片方式

1. **客户端分片**：客户端使用**一致性哈希等算法**决定**键**应当**分布到哪个节点**。（Redis主从，哨兵）。
2. **代理分片**：将客户端**请求发送到代理**上，由**代理转发**请求到**正确的节点**上。
3. **服务器分片**：**Redis Cluster**。

## 三、Hash Tag（Cluster）

在 Redis Cluster 环境下，**Hash Tag（哈希标签）** 是解决跨槽（Cross-Slot）操作受限的终极武器。

当在集群模式下使用 `MSET`、`MGET` 或者执行 Lua 脚本时，如果涉及的多个 Key 分布在不同的哈希槽（Slot）中，Redis 为了保证性能和一致性，会直接报错：`(error) CROSSSLOT Keys in request don't hash to the same slot`。

Hash Tag 的出现就是为了打破这个限制，允许你**强制将一批相关的 Key 路由到同一个哈希槽中**。

```bash
# {user} 为 hash tag，同一 tag 的键在同一槽位
SET {user}:1001:name "Alice"
SET {user}:1001:age "25"
# 可对这两个键进行多键操作
MGET {user}:1001:name {user}:1001:age
```

---

### 1. Hash Tag 的语法规则

Hash Tag 的语法非常简单：**在 Key 中使用一对花括号 `{}` 将特定字符包裹起来。**

**核心计算逻辑：**
当 Redis Cluster 计算一个 Key 的哈希槽时，如果发现 Key 中包含 `{}`，它将**不再使用整个 Key 名进行哈希计算，而是仅仅使用 `{}` 内部的字符串进行 `CRC16` 计算。**

* 如果 Key 中有多个 `{}`，Redis 只认**第一对**。
* 如果 `{}` 内部是空的（例如 `user:{}name`），Redis 会退化为使用整个 Key 进行计算。

---

### 2. 具体用法演示与对比

我们来模拟一个典型的业务场景：你需要同时缓存一个用户的“基本信息”和该用户的“最新订单状态”，并且希望用 `MSET` 一次性写入以提高效率。

#### 1). 不使用 Hash Tag（演示失败场景）

假设我们设计的 Key 分别是 `user:1000:profile` 和 `user:1000:order`。

```bash
# 尝试使用 MSET 批量写入
127.0.0.1:6379> MSET user:1000:profile "Alice" user:1000:order "paid"
(error) CROSSSLOT Keys in request don't hash to the same slot

```

**失败原因：** `user:1000:profile` 和 `user:1000:order` 的完整字符串不同，`CRC16(key) % 16384` 算出来的槽位大概率不在同一个 Master 节点上，Redis 拒绝跨节点事务。

#### 2). 使用 Hash Tag（演示成功场景）

我们将 Key 的设计稍微修改一下，提取它们共同的标识（比如用户 ID），用 `{}` 包裹起来：`{user:1000}:profile` 和 `{user:1000}:order`。

```bash
# 再次尝试使用 MSET
127.0.0.1:6379> MSET {user:1000}:profile "Alice" {user:1000}:order "paid"
OK

# 尝试使用 MGET 批量读取
127.0.0.1:6379> MGET {user:1000}:profile {user:1000}:order
1) "Alice"
2) "paid"

```

**成功原因：** Redis 在计算这两个 Key 的槽位时，发现都有 `{}`，于是**只对 `user:1000` 这个字符串进行哈希计算**。因为输入内容完全一致，所以它们必然被分配到同一个哈希槽，落入同一个 Master 节点。跨槽限制随之解除。

---

### 3. 避坑指南：数据倾斜（Data Skew）

Hash Tag 虽然好用，但它是一把**双刃剑**。在使用时，你必须警惕**数据倾斜**问题。

Redis Cluster 的核心设计初衷是“打散数据，负载均衡”。如果你滥用 Hash Tag，会导致大量的数据被硬塞进同一个槽位，最终落到同一台物理机器上。

#### **反面教材：**

假设你为了图方便，把所有商品相关的缓存都加上了同一个全局 Tag，比如 `{global_goods}:phone:1`、`{global_goods}:laptop:2`……
这样做的后果是，不管你有多少万个商品，它们都会被计算到**同一个哈希槽**里。集群中负责这个槽位的那台 Master 节点会被瞬间挤爆（内存飙升、CPU 满载），而其他 Master 节点却在闲置。这就完全失去了集群分片扩容的意义。

#### **最佳实践：**

Hash Tag 里面的颗粒度一定要足够细。通常建议以 **User ID、Order ID 或者 Session ID** 作为 Hash Tag 的内容。这样既保证了单个用户的相关数据能一起操作，又保证了全局宏观上不同用户的数据能均匀打散在整个集群中。

### 4.适用场景

- 数据量超过单机内存限制
- 高并发读写场景
- 需要水平扩展的场景
- 对高可用要求较高的生产环境
