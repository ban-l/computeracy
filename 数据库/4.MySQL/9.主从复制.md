# 主从复制

主从复制/读写分离：提高性能，冗余服务器，数据备份，高可用。

这是分布式数据库架构的基石。理解这个流程对于解决 **“读写分离后的数据不一致”** 问题至关重要。

简单来说，MySQL 的主从复制就是 **“一台服务器（Master）负责写，然后把改动记录（Binlog）传给另一台（Slave），后者照着重做一遍”**。

这个过程主要由 **三个核心线程** 和 **两个日志** 协作完成。

1. Binlog 线程 ：负责将Master服务器上的数据更改写入二进制日志（Binary  log）中。
2. I/O 线程 ：负责从Master服务器上读取二进制日志，并写入Slave服务器的中继日志（Relay log）。
3. SQL 线程 ：负责读取中继日志，解析出Master服务器已经执行的数据更改并在Slave服务器中重放（Replay）。

## 一 主从复制过程

------

### 1. 核心流程详解 (The Three-Thread Model)

我们把这个过程拆解为 5 个步骤，涵盖了从 Master 到 Slave 的全链路：

#### 第一步：Master 记录变更 (Binlog)

当你在 Java 应用中执行 `INSERT/UPDATE/DELETE` 并提交事务时：

1. Master 按照我们在上一节提到的“两阶段提交”，将修改记录顺序写入 **Binlog**。
2. **注意**：Binlog 写完，Master 的主任务就结束了，它不需要等 Slave。

#### 第二步：Slave 发起连接 (I/O Thread)

Slave 上会运行一个 **I/O 线程**。

1. 它连接到 Master。
2. 发送请求：“请把 `mysql-bin.000001` 文件中，位置 `890` 之后的日志发给我。”

#### 第三步：Master 发送日志 (Bin Dump Thread)

Master 接收到请求后，会专门启动一个 **Binlog Dump 线程**。

1. 它负责读取本地的 Binlog。
2. 通过网络将日志流发给 Slave。

#### 第四步：Slave 接收并写入中继日志 (Relay Log)

Slave 的 I/O 线程收到数据后，**并不会立即执行 SQL**。

1. 它把收到的二进制日志写入本地的一个临时文件，叫 **Relay Log (中继日志)**。
2. 写完后，I/O 线程的任务就完成了（只负责搬运）。

#### 第五步：Slave 重放日志 (SQL Thread)

Slave 上还有一个 **SQL 线程**（在 MySQL 5.6 之前是单线程，之后支持多线程）。

1. 它实时监控 Relay Log。
2. 发现有新内容，就读出来，解析成 SQL 语句。
3. 在 Slave 的数据库里**再执行一遍**。
4. 执行成功后，Slave 的数据就和 Master 一致了。

## 二 复制的三种模式 (从“快”到“稳”)

作为架构师，你需要根据业务对数据一致性的要求选择模式：

| **模式**                   | **描述**                                                     | **优点**                         | **缺点**                                    |
| -------------------------- | ------------------------------------------------------------ | -------------------------------- | ------------------------------------------- |
| **异步复制 (默认)**        | Master 写完 Binlog 就返回成功，不管 Slave 有没有收到。       | **性能最好**。                   | 主库宕机时，**Slave 可能丢失刚才的数据**。  |
| **半同步复制 (Semi-Sync)** | Master 写完 Binlog，必须等**至少一个** Slave 收到并写入 Relay Log，才返回成功。 | **数据安全性高**，基本不丢数据。 | 性能有损耗（取决于网络延迟）。              |
| **全同步复制 (MGR)**       | 所有节点都确认收到才算成功。                                 | 数据绝对一致。                   | 性能极差，一般不怎么用（除非用 MGR 集群）。 |

**半同步复制**

- 主库写入 binlog 日志后，强制立即将数据同步到从库；

- 从库将日志写入自己本地的 relay log 之后，接着会返回一个 ack 给主库；
- 主库接收到至少一个从库的 ack 之后认为写操作完成了。

## 三 主从延迟 (Replication Lag)

这是读写分离架构中最大的坑。

**现象**：用户刚注册（写 Master），立马登录（读 Slave），结果提示“账号不存在”。因为 Slave 的 SQL 线程还没来得及执行那条 Insert。

**原因分析**：

1. **大事务**：Master 上执行了一个 `DELETE` 耗时 3 秒，Slave 重放时也得花 3 秒，这期间就延迟了。
2. **网络抖动**：Binlog 传得慢。
3. **单线程瓶颈**：早期 Slave 只有一个 SQL 线程，Master 并发写了几千条，Slave 只能一条条排队执行。

**解决方案 (Java 侧)**：

1. **强制路由**：对于即时性要求极高的读取（如注册后立马登录、支付后查看订单），在代码里强制去 **Master** 读。
2. **判断延迟**：在读 Slave 之前，先检查 `Seconds_Behind_Master` 参数，如果延迟太大，自动降级读 Master。

## 四 读写分离

主服务器处理写操作、实时性要求比较高的读操作，从服务器处理读操作。

读写分离能提高性能的原因在于：

- 主从服务器负责各自的读和写，极大程度缓解了锁的争用；
- 从服务器可以使用 MyISAM，提升查询性能、节约系统开销；
- 可以动态地调整从服务器的数量，从而调整整个数据库的性能；
- 冗余服务器，数据备份，高可用，提高数据安全。

**读写分离常用代理方式来实现**：代理服务器接收应用层传来的读写请求，然后决定转发到哪个服务器。

## 五 总结

主从复制的本质是 **Binlog 的搬运与重放**。

- **Master**：只管写 Binlog，通过 Dump 线程发送。
- **Slave**：
  - **I/O 线程**：负责把数据搬回家（Relay Log）。
  - **SQL 线程**：负责把数据消化掉（Execute）。
