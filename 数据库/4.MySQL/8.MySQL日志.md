# MySQL 日志

MySQL 的日志系统主要由“三剑客”组成：**Binlog**（归档日志）、**Redo Log**（重做日志）和 **Undo Log**（回滚日志）。

------

## 1. Binlog (归档日志) —— 公司的“永久档案”

- **位置**：MySQL Server 层（所有存储引擎共用）。
- **性质**：**逻辑日志**。它记录的是语句的原始逻辑，比如“给 ID=2 的行，age 字段加 1”。
- **作用**：
  1. **主从复制 (Replication)**：Master 把 Binlog 发给 Slave，Slave 重放一遍，实现数据同步。
  2. **数据恢复**：如果你误删了库，可以通过 Binlog 找到删除前的时间点，把数据恢复回去。
- **特点**：**追加写**。写满一个文件就创建一个新的，不会覆盖旧数据。

------

## 2. Redo Log (重做日志) —— 掌柜的“粉板” (InnoDB 特有)

这是 InnoDB 引擎为了解决“高性能”和“不丢数据”矛盾而发明的神器。它保证了事务的 **D (Durability, 持久性)**。

- **痛点**：如果每次更新都直接写磁盘（随机 I/O），速度太慢。
- **方案 (WAL 技术)**：全称 **Write-Ahead Logging**（先写日志，再写磁盘）。
  - 当有一条记录需要更新时，InnoDB 会先把记录写到 Redo Log（顺序 I/O，速度极快），并更新内存，这就算更新完成了。
  - 空闲时，InnoDB 会在后台慢慢把 Redo Log 里的操作刷到磁盘的数据文件里（这叫“刷脏页”）。
- **结构**：**循环写**。Redo Log 的空间是固定的（比如 4 个文件，每个 1GB）。
  - `write pos`：当前记录的位置，一边写一边后移。
  - `checkpoint`：当前要擦除的位置（已经刷到磁盘的数据），也是往后推。

> **专家视角的 Crash Safe**：
>
> 如果 MySQL 突然宕机，内存里的数据还没来得及刷到磁盘。没关系，重启后，InnoDB 会去读取 Redo Log，发现有记录没同步到磁盘，就重新执行一遍。这就是 **Crash Safe** 能力。

------

## 3. Undo Log (回滚日志) —— 后悔药 (InnoDB 特有)

它保证了事务的 **A (Atomicity, 原子性)** 和 **I (Isolation, 隔离性)**。

- **作用**：
  1. **事务回滚**：你执行 `UPDATE` 改了数据，但还没 Commit 就要 Rollback。数据库怎么知道原来是什么？全靠 Undo Log。它记录了**相反**的操作：
     - 你 `INSERT`，它记 `DELETE`。
     - 你 `UPDATE` 把 1 改成 2，它记 `UPDATE` 把 2 改成 1。
  2. **MVCC (多版本并发控制)**：当读取记录时，若该记录被其他事务占用，可以通过 Undo Log 读取之前的版本（快照读）。

------

## 4. 深度解析：一条 Update 语句的“两阶段提交”

这是面试中最具区分度的问题：**既然有了 Redo Log 保证不丢，为什么还要 Binlog？两者怎么配合？**

假设执行：`UPDATE T SET c=c+1 WHERE ID=2;`

为了让两份日志逻辑一致（避免主从数据不一致），MySQL 使用了 **两阶段提交 (Two-Phase Commit)**。

**执行流程：**

1. **执行器**：找引擎取 ID=2 这一行。
2. **InnoDB**：拿到数据，修改内存，写入 **Redo Log**，此时 Redo Log 处于 **`PREPARE`** 状态。告诉执行器：“我准备好了”。
3. **执行器**：生成并在磁盘写入 **Binlog**。
4. **执行器**：调用引擎的提交事务接口。
5. **InnoDB**：把 Redo Log 改成 **`COMMIT`** 状态。更新完成。

**为什么要这么麻烦？（反证法）**

- **先写 Redo Log 再写 Binlog**：如果写完 Redo Log 机器挂了，Binlog 没写。重启后主库通过 Redo Log 恢复了数据，但从库因为没收到 Binlog 导致丢失更新。**主从不一致。**
- **先写 Binlog 再写 Redo Log**：如果写完 Binlog 机器挂了，Redo Log 没写。主库重启后这条更新丢了（因为没 Redo Log），但从库收到了 Binlog 并执行了更新。**主从不一致。**

**只有两阶段提交，才能保证两份日志逻辑上的一致性。**

------

## 5. 专家总结表

| **特性**     | **Binlog (归档日志)**  | **Redo Log (重做日志)** | **Undo Log (回滚日志)** |
| ------------ | ---------------------- | ----------------------- | ----------------------- |
| **所属层级** | Server 层 (通用)       | InnoDB 引擎层           | InnoDB 引擎层           |
| **记录内容** | 逻辑 SQL (语句/行数据) | 物理修改 (页偏移量)     | 逻辑反向操作            |
| **写入方式** | 追加写 (无限增长)      | 循环写 (空间固定)       | -                       |
| **核心职责** | 主从复制、历史恢复     | **Crash Safe (持久性)** | **事务回滚 (原子性)**   |
| **存在时间** | 永久保存               | 刷盘后可覆盖            | 事务结束后按需删除      |

## 给 Java 工程师的实战建议

1. **双 1 配置**：

   在生产环境中，为了保证数据绝对不丢，务必将以下两个参数设为 1：

   - `sync_binlog = 1` (每次提交事务都强制刷 Binlog 到磁盘)
   - `innodb_flush_log_at_trx_commit = 1` (每次提交事务都强制刷 Redo Log 到磁盘)
   - *代价是 IO 性能会下降，但在金融级业务中是必须的。*

2. **长事务的危害**：

   如果一个事务开了很久不提交，Undo Log 就无法清理（因为可能还需要回滚或供 MVCC 使用），导致系统空间迅速膨胀。所以在 Java 代码中，`@Transactional` 的范围越小越好。
