# 索引下推

如果说 B+ 树的结构决定了“怎么找数据”，那么 **ICP (Index Condition Pushdown)** 就决定了“怎么用好已有的数据”。

它是 MySQL 5.6 引入的一个重大改进，**核心目的是：减少“回表”次数，从而减少磁盘 I/O。**

联合索引：**`idx_user (name, age)`**，场景 SQL：

```sql
SELECT * FROM users WHERE name LIKE '张%' AND age = 10;
```

*(注意：这里 `name` 使用了模糊查询，根据最左前缀原则，索引在 `name` 这里就断了，`age` 本来是无法用来定位的。)*

---

## 1. 没有 ICP 的世界 (MySQL 5.5 及之前)

在没有索引下推的时候，架构是这样的：**存储引擎（InnoDB）很“笨”，Server 层很“累”**。

**执行流程：**

1. **Server 层**告诉 InnoDB：“给我所有 `name` 是 '张' 开头的数据。”
2. **InnoDB** 在联合索引里找到第一条 '张三'（此时索引里虽然有 `age=20`，但 InnoDB 视而不见，因为它只接到了 `name` 的指令）。
3. **InnoDB** 拿着主键 ID，去聚簇索引里**回表**，把 '张三' 的完整数据行捞出来，扔给 Server 层。
4. **Server 层** 拿到完整数据，判断 `age` 是否等于 10。发现是 20，**扔掉**。
5. **InnoDB** 继续找下一条 '张四'（索引里 `age=10`）。
6. **InnoDB** 再次**回表**，捞出 '张四' 的完整数据，扔给 Server 层。
7. **Server 层** 判断 `age` 是否等于 10。是，**保留**。

**痛点：**
虽然索引里明明有 `age` 的值，但 InnoDB 却没利用，导致了大量的**无效回表**（比如 '张三' 明明不符合 age=10，却还是跑了一趟磁盘读完整数据）。

---

## 2. 有了 ICP 的世界 (MySQL 5.6 及之后)

现在，Server 层把**过滤条件“下推”**给了存储引擎。

**执行流程：**

1. **Server 层**告诉 InnoDB：“给我所有 `name` 是 '张' 开头的数据，**同时，如果 `age` 不等于 10，你就别给我了**。”
2. **InnoDB** 在联合索引里找到 '张三'。它看了一眼索引节点里的 `age`，发现是 20。
3. **InnoDB 直接跳过**（不回表了！），继续找下一个。
4. **InnoDB** 找到 '张四'，看一眼索引里的 `age`，发现是 10。
5. **InnoDB** 拿着主键 ID **回表**，捞出完整数据，扔给 Server 层。
6. **Server 层** 再次校验（双重保险），返回结果。

**收益：**
在 `name LIKE '张%'` 匹配了 10 万人，但只有 1 个人是 `age=10` 的极端情况下：

* **无 ICP**：回表 10 万次。
* **有 ICP**：回表 1 次。
* 性能提升可能是 **100 倍**。

---

## 3. ICP 的生效条件

并不是所有查询都能触发 ICP，必须满足以下条件：

1. **必须是联合索引**：如果是单列索引，过滤完了也没东西可推了。
2. **必须是二级索引**：聚簇索引（主键）本身就存了所有数据，不存在“回表”这一说，所以不需要 ICP。
3. **针对的是“范围查询”或“模糊匹配”导致索引失效后的后续列**：

   * `WHERE name = 'Alice' AND age = 10`：这个不需要 ICP，因为按照最左前缀，`age` 本身就参与定位了。

   * `WHERE name LIKE 'A%' AND age = 10`：**这是 ICP 的主战场**。

---

## 4. 实战：如何验证 ICP 正在工作？

作为 Java 工程师，你在调优时需要使用 `EXPLAIN` 命令。

查看 `Extra` 列：

* **Using index condition**：恭喜你，ICP 生效了。
* **Using where**：说明没用上 ICP，可能发生了全量回表。

```sql
EXPLAIN SELECT * FROM users WHERE name LIKE '张%' AND age = 10;
```

| id | table | type | key | Extra |
| --- | --- | --- | --- | --- |
| 1 | users | range | idx_name_age | **Using index condition** |

---

## 总结

**索引下推 (ICP)** 的本质，就是让存储引擎“多干点活”（在索引遍历阶段就把不符合条件的数据过滤掉），从而减少“回表”这个昂贵的磁盘 I/O 操作。

这是一个典型的 **“空间（索引里已经存了值）换 时间（IO）”** 的优化。
