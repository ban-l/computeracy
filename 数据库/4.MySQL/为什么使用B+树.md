## 为什么使用 B+ 树索引

核心原则只有一条：**数据库的性能瓶颈在于磁盘 I/O。谁能用最少的磁盘读取次数找到数据，谁就是赢家。**

------

## 一 B+ 树 VS 哈希表 (Hash Table)

哈希表是所有数据结构中查找速度最快的，时间复杂度为 `O(1)`。Redis 的核心就是 Hash。那为什么 MySQL 不用它做默认索引？

- **痛点**：**哈希表不支持“范围查询”和“排序”。**
- **场景**：
  - `SELECT * FROM users WHERE id = 1;` —— Hash 很快。
  - `SELECT * FROM users WHERE age > 18;` —— Hash 彻底傻眼。因为它存的数据是散列无序的，无法知道谁比 18 大，只能全表扫描。
- **结论**：MySQL 是通用数据库，大量业务依赖 `>`、`<`、`BETWEEN` 和 `ORDER BY`，所以 Hash 被淘汰（虽然 Memory 引擎支持，InnoDB 自适应哈希索引也用到，但不能做主索引）。

------

## 二 B+ 树 VS 二叉树 / 红黑树 (Binary Tree)

我们在 Java `HashMap` 或 `TreeMap` 中常用的红黑树，在内存中表现极佳，但在磁盘上就是灾难。

- **痛点**：**树太“高”了 (Height)。**
- **原理**：二叉树每个节点只能分两个叉。如果你有 1000 万行数据，树的高度会达到 20 层左右 (`2^24 ♒︎ 1600万`)。
- **I/O 代价**：在最坏情况下，为了查一行数据，你需要读取 20 次磁盘（每次读取一个节点）。对于机械硬盘，这需要 0.2 秒；即使是 SSD，并发高了也扛不住。
- **结论**：我们需要一种“矮胖”的树，而不是“高瘦”的树。

------

## 三 B+ 树 VS B 树 (B-Tree) —— 关键对决

B 树（多路平衡查找树）是为了解决磁盘 I/O 而生的。它允许一个节点有多个子节点（比如 1000 个叉），这让树的高度瞬间降到了 3 层左右。

但 InnoDB 依然选择了 B 树的改良版 —— **B+ 树**。这是为什么？

### 区别 1：非叶子节点不存数据 (核心优势)

- **B 树**：每个节点（包括根节点、中间节点）都存储 key 和 **value（实际数据行）**。
- **B+ 树**：只有叶子节点存储 **value（实际数据行）**，非叶子节点**只存 key (索引)** 和指针。

> **专家解读**：
>
> 数据库的数据页大小是固定的（InnoDB 默认 16KB）。
>
> - 如果你用 **B 树**，一行数据假如占 1KB，那一个 16KB 的节点只能存 16 条数据，分叉（Fan-out）就很少，树就会变高。
> - 如果你用 **B+ 树**，中间节点不存数据，只存主键 ID（假设 8字节）和指针（6字节）。16KB 的空间可以存 `16KB / 14B ♒︎1170`个指针！
>
> **结果**：B+ 树变得**更矮、更胖**。通常 3 层高度的 B+ 树就能存 2000 万行数据（`1170 * 1170 * 16`），查询只需要 3 次 I/O（甚至更少，因为根节点常驻内存）。

### 区别 2：叶子节点的链表结构 (范围查询杀手锏)

- **B 树**：叶子节点之间是独立的。如果要进行范围查询（`> 18`），你需要像中序遍历二叉树一样，在树的各层之间跳来跳去。
- **B+ 树**：所有叶子节点通过**双向链表**连接在一起。
- **场景**：`SELECT * FROM users WHERE id > 100;`
  - B+ 树只需要找到 id=100 的那个叶子节点，然后顺着链表向后读取即可。这也是为什么数据库通常通过主键分页效率很高的原因。

### 区别 3：查询稳定性

- **B 树**：运气好在根节点就找到了（1次 I/O），运气不好在叶子节点才找到（3次 I/O）。速度不稳定。
- **B+ 树**：无论找什么数据，都必须走到最后以一层的叶子节点。所有查询的耗时是极其稳定的。

------

## 总结 为什么是 B+ 树

InnoDB 选择 B+ 树不是因为它是最快的，而是因为它是**最适合磁盘存储**和**SQL 业务场景**的：

1. **极少的磁盘 I/O**：通过“瘦身”非叶子节点，让树高保持在 2-4 层，即使数据量达到亿级。
2. **吞吐量高**：一次 I/O（读取一页）能载入更多的索引键值。
3. **擅长范围查询**：叶子节点的双向链表完美契合 SQL 的 `ORDER BY` 和范围查找需求。
4. **全表扫描快**：B 树做全表扫描需要遍历整棵树，B+ 树只要遍历底层的链表。

总结

1. B+ 树有更低的树高；
2. 出度大，一个索引节点指向更多的数据：
   1. 高度低，寻道时间少，磁盘读写代价更低；
   2. 操作系统一般将内存和磁盘分割成固定大小的块，每一块称为一页，内存与磁盘以页为单位交换数据；
   3. 数据库系统将索引的一个节点的大小设置为页的大小，使得一次 I/O 就能完全载入一个节点；
   4. 如果数据不在同一个磁盘块上，那么通常需要移动制动手臂进行寻道，寻道会增加磁盘数据读取时间；
   5. B+树的⾮叶⼦节点可以存放更多的索引，查询底层节点的磁盘 I/O次数会更少。
3. 磁盘预读特性：
   1. 为了减少磁盘 I/O 操作，磁盘往往不是严格按需读取，而是每次都会预读；
   2. 预读过程中，磁盘进行顺序读取，顺序读取不需要进行磁盘寻道，并且只需要很短的磁盘旋转时间，速度会非常快；
   3. 并且可以利用预读特性，相邻的节点也能够被预先载入。
4. B+树的数据在叶子结点中，分支结点均为索引，查找时只需要扫一遍叶子结点；
5. 插⼊和删除效率：B+ 树有⼤量的冗余节点，删除⼀个节点的时候，可以直接从叶⼦节点中删除，甚⾄可以不动⾮叶⼦节点，删除⾮常快。
