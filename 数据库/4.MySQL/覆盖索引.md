# 覆盖索引

 **覆盖索引 (Covering Index)** 目标——**消灭回表**。

这是 SQL 优化中**性价比最高**的手段。如果你的查询能命中覆盖索引，查询速度通常会有数量级的提升。

---

## 1. 核心概念：什么是“覆盖”？

**二级索引**（非聚簇索引）的叶子节点里，存储的是 **索引列 + 主键 ID**。

* **普通查询（需要回表）**：
`SELECT * FROM users WHERE name = 'Alice';`
虽然 `name` 索引找到了 Alice，但你想要 `*`（包括 city, gender 等），索引里没有。所以 InnoDB 必须拿着 ID 回到主键索引（聚簇索引）里把整行数据捞出来。
* **覆盖索引（无需回表）**：
`SELECT id, name FROM users WHERE name = 'Alice';`
你要查的 `id` 和 `name`，刚好都在这个二级索引树的节点里！
**InnoDB 发现：“哎？你要的东西我这儿全都有，不用去查主键树了。”** 直接返回数据。

---

## 2. 实战演练：如何触发？

假设我们有一个表 `users`，主键是 `id`，有一个联合索引 **`idx_name_age (name, age)`**。

### 场景 A：完美的覆盖

```sql
SELECT name, age FROM users WHERE name = 'Alice';
```

* **分析**：查询条件是 `name`（在索引里），需要的字段是 `name` 和 `age`（全在索引里）。
* **结果**：**触发覆盖索引**。速度极快。

### 场景 B：隐式的覆盖 (Expert Trick)

```sql
SELECT id, name, age FROM users WHERE name = 'Alice';
```

* **分析**：多查了一个 `id`。
* **原理**：请记住，**InnoDB 的二级索引叶子节点天然包含主键 ID**。所以 `id`、`name`、`age` 三个字段都在这棵树上。
* **结果**：**触发覆盖索引**。

### 场景 C：功亏一篑

```sql
SELECT name, age, city FROM users WHERE name = 'Alice';
```

* **分析**：多查了一个 `city`。
* **结果**：索引里没有 `city`，必须**回表**。性能瞬间下降。

---

## 3. 如何验证？(EXPLAIN)

依然是看执行计划。这是判断是否覆盖最权威的标准。

执行：

```sql
EXPLAIN SELECT name, age FROM users WHERE name = 'Alice';
```

关注 **`Extra`** 这一列：

* **Using index**：恭喜！这代表用到了覆盖索引（不需要回表）。
* **NULL** 或 **Using index condition**：代表虽然用了索引，但还是回表了（或者用了 ICP）。

> **注意区分**：
> * `Using index`：**最高境界**。全程只读索引树，不需要读数据行。
> * `Using index condition`：**ICP**。用了索引过滤，但还得回表取数据。
>

## 4. 覆盖索引的“超纲”应用：解决 `LIKE '%abc'` 失效问题

我们都知道 `LIKE '%张'`（左模糊）会导致索引失效，变成全表扫描。
**但是，如果是覆盖索引，它能救命！**

```sql
-- 假设只有 id, name, age 三列
SELECT id, name, age FROM users WHERE name LIKE '%张%';
```

* **逻辑**：
1. 虽然不能通过 B+ 树快速定位（因为是 % 开头）。
2. 但是，优化器发现：**“你要的所有数据，都在 `idx_name_age` 这个索引树上，这棵树比全表数据（聚簇索引）要小得多！”**
3. **决策**：优化器决定**扫描整个索引树**（Index Full Scan），而不是扫描全表。


* **性能**：虽然也是全扫描，但扫描“目录”肯定比扫描“厚书”要快得多。

---

## 5. 建议：如何利用覆盖索引优化设计？

1. 拒绝 `SELECT *`：只要多查了一个无关字段（比如 `create_time`），覆盖索引就直接失效。
2. **针对高频查询设计联合索引**；
3. **大分页优化 (Late Row Lookups)**：
当 `LIMIT 1000000, 10` 这种深分页出现时，回表消耗巨大。
**优化方案**：
```sql
-- 1. 先在子查询里利用覆盖索引，只查出 ID (无需回表，快)
SELECT id FROM users WHERE age > 20 LIMIT 1000000, 10

-- 2. 再拿这 10 个 ID 去关联主表 (只回表 10 次)
SELECT * FROM users u
JOIN (SELECT id FROM users WHERE age > 20 LIMIT 1000000, 10) t
ON u.id = t.id;
```

---

## 总结

* **核心收益**：**0 回表**。
* **适用场景**：查询列（Select List）是索引列的子集。
* **代价**：索引不能太宽（太多列），否则索引文件过大，反而影响内存利用率。
