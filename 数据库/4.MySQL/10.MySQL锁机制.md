# MySQL 的锁机制

MySQL 锁从外到里分为三层：**全局锁 -> 表级锁 -> 行级锁**。

------

## 一 全局锁 (Global Lock) —— “关门打烊”

- **命令**：`Flush tables with read lock (FTWRL)`
- **效果**：整个数据库实例变成**“只读”**状态。所有的写操作（增删改）、建表修改表结构，甚至更新事务，全部被阻塞。
- **场景**：极其少见。通常只用于**全库逻辑备份**（mysqldump），确保备份期间没有新数据写入，保证数据一致性。
- **替代方案**：现在的 `mysqldump` 配合 `--single-transaction` 参数（利用 MVCC），可以在不锁库的情况下进行备份。所以，**尽量别用全局锁**。

------

## 二 表级锁 (Table Lock) —— “锁住整个房间”

这是 MyISAM 的标配，但在 InnoDB 中也存在。

### 1. 表锁 (Table Lock)

- **命令**：`LOCK TABLES t1 READ/WRITE`
- **特点**：开销小，加锁快；不会出现死锁；但锁粒度大，并发度最低。
- **Java 场景**：在 InnoDB 中很少显式使用。除非你在做某些特殊的跑批任务，需要手动独占一张表。

### 2. 元数据锁 (MDL - Metadata Lock) —— **隐形杀手**

这是你必须警惕的。MDL 不需要你显式去加，MySQL 自动加。

- **触发机制**：
  - 当你做 **增删改查 (DML)** 时，加 **MDL 读锁**。
  - 当你做 **修改表结构 (DDL)**（比如 `ALTER TABLE` 加字段）时，加 **MDL 写锁**。
- **互斥规则**：读锁之间不互斥，但**读写互斥**、**写写互斥**。
- **线上事故经典场景**：
  1. 事务 A 正在查询（加了 MDL 读锁），但是事务没提交（可能查询很慢，或者 Java 端处理耗时）。
  2. 此时，DBA 或者是你在发布版本时，执行了一个 `ALTER TABLE` 加字段（试图加 MDL 写锁）。
  3. **阻塞**：因为 MDL 读写互斥，`ALTER` 被卡住。
  4. **灾难**：后续所有进来的查询（SELECT），因为要等 MDL 写锁释放，全都被卡住！瞬间连接池爆满，服务挂掉。

------

## 三 行级锁 (Row Lock) —— “精准打击” (InnoDB 核心)

InnoDB 相比 MyISAM 的最大优势就是支持行锁。

**但请记住一句金科玉律：InnoDB 的行锁是锁在“索引”上的，而不是锁在物理行上。** 如果查询没有命中索引，行锁就会退化为表锁。

### 1. 锁的读写属性 (Shared vs Exclusive)

- **共享锁 (S锁 - Shared Lock)**：即“读锁”。
  - *含义*：我读的时候，允许你也读，但不许你改。
  - *Java 写法*：`SELECT ... LOCK IN SHARE MODE;`
- **排他锁 (X锁 - Exclusive Lock)**：即“写锁”。
  - *含义*：我改的时候，你既不能读（加锁读），也不能改。
  - *Java 写法*：`SELECT ... FOR UPDATE;` (以及自动加锁的 `UPDATE`, `DELETE`, `INSERT`)

### 2. 锁的算法 (The Algorithms) 

在 **Repeatable Read (可重复读)** 隔离级别下（MySQL 默认），为了解决“幻读”问题，InnoDB 引入了三种复杂的锁算法。

假设我们要操作一张表 `t`，只有一列 `id`，已有的值为：`1, 5, 10`。

- **Record Lock (记录锁)**：
  - 只锁住**一行记录**。
  - *触发*：精准命中唯一索引。
  - *SQL*：`SELECT * FROM t WHERE id = 5 FOR UPDATE;` -> 锁住 `id=5` 这一行。
- **Gap Lock (间隙锁)**：
  - 锁住**两个记录之间的空隙**（不包含记录本身），防止别人在这个空隙里插入新数据（防止幻读）。
  - *触发*：范围查询，或者查询不存在的记录。
  - *SQL*：`SELECT * FROM t WHERE id = 6 FOR UPDATE;`
  - *效果*：因为 6 不存在，InnoDB 会锁住 `(5, 10)` 这个开区间。此时，如果有另一个事务想 `INSERT INTO t VALUES(7);`，会被阻塞！
- **Next-Key Lock (临键锁)**：
  - **Record Lock + Gap Lock** 的组合。锁住记录本身 + 记录前面的间隙。
  - **InnoDB 的默认行锁算法**。
  - *区间*：前开后闭 `(5, 10]`。
  - *SQL*：`SELECT * FROM t WHERE id > 5 AND id < 10 FOR UPDATE;`
  - *效果*：锁住 `(5, 10]`。

## 四 意向锁 (Intention Lock) —— 门上的红灯

这是一种**表级锁**，分为 `IS` (意向共享) 和 `IX` (意向排他)。

很多同学不理解为什么需要它。

- **场景**：事务 A 锁住了表中的**一行** (`id=1`)。此时事务 B 想去锁住**整张表** (`LOCK TABLES ... WRITE`)。
- **冲突检测**：如果没有意向锁，事务 B 就必须去遍历整张表，看看到底有没有哪一行被锁住了。这效率太低了！
- **机制**：事务 A 在锁行之前，先在“表门口”贴个条子（意向锁 IX）：“里面有人在干活”。事务 B 一看门口有条子，就知道这表不能锁，直接等待。
- **结论**：意向锁是 MySQL 内部自动协调**行锁与表锁冲突**的机制，不需要你操心。

------

## 五 避坑指南

1. **更新务必走索引**：

   `UPDATE users SET age = 30 WHERE name = 'Steve';`

   如果 `name` 字段没有索引，InnoDB 会**锁住整张表**的所有记录（甚至包括不存在的间隙），导致并发能力归零。

2. **警惕死锁 (Deadlock)**：

   - *场景*：事务 A 锁了 ID=1，想要锁 ID=2；事务 B 锁了 ID=2，想要锁 ID=1。
   - *日志*：如果你在 Java 日志里看到 `Deadlock found when trying to get lock`。
   - *解决*：确保所有事务按照**相同的顺序**去操作资源（比如都先锁 ID 小的，再锁 ID 大的）。

3. **间隙锁的副作用**：

   在并发插入非常高的场景（比如发号器、流水号生成），如果使用了范围条件的 `FOR UPDATE`，会导致大量的 Gap Lock 竞争，严重降低吞吐量。这时候可以考虑把隔离级别降级为 **Read Committed (RC)**，RC 级别下没有间隙锁。
